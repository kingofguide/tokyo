# 🎲 King of Guide — Tokyo Monsters

A fan-made web guide for King of Tokyo monsters. Built with plain HTML/CSS/JS and Markdown content files.

## Project Structure
```
tokyo/
├── monsters/                ← One .md file per monster
│   ├── alienoid.md
│   ├── cyber-kitty.md
│   ├── gigazaur.md
│   ├── meka-dragon.md
│   ├── space-penguin.md
│   └── the-king.md
├── build.py                 ← Converts .md files → data.json
├── data.json                ← Generated by build.py (don't edit manually)
├── index.html               ← The web app (no framework, just HTML/CSS/JS)
└── README.md                ← This file
```

## Workflow: Adding or Editing a Monster

### 1. Create (or edit) a Markdown file in `/monsters`

Copy an existing file and rename it (e.g. `cyber-bunny.md`). The filename becomes the monster's ID.

Each file has two parts:

**Frontmatter** (between the `---` lines) — structured data:
```yaml
---
name: Cyber Bunny
icon: 🐰
tagline: "Fast, sneaky, impossible to pin down."
difficulty: Intermediate       # Beginner | Intermediate | Advanced
type: Mutant                   # Beast | Alien | Mutant | Robot
game:
  - Tokyo
style:
  - Smasher
  - Scorer
stats:
  aggression: 3
  resilience: 2
  energy: 4
  speed: 5
---
```

**Body** — free-form Markdown (headings, lists, bold text, blockquotes etc.):
```markdown
## Overview
Cyber Bunny is all about speed...

## Evolution Cards
### Some Card *(Permanent)*
> *Official card text goes here.*

Commentary on the card goes here.
```

### 2. Run the build script
```bash
python build.py
```

This reads all `.md` files and generates `data.json`. Run this every time you change a monster file.

### 3. Preview locally

Use a local server to preview the site:
```bash
python -m http.server 8080
# then open http://localhost:8080
```

### 4. Commit and push to GitHub
```bash
git add .
git commit -m "Add or update monster"
git push
```

Make sure `data.json` is committed — it is required at runtime.

## Monster File Reference

### Required frontmatter fields

| Field       | Example                      | Notes                              |
|-------------|------------------------------|------------------------------------|
| `name`      | `King Lizard`                | Display name                       |
| `icon`      | `🦎`                         | Emoji shown in the monster slider  |
| `tagline`   | `"The classic destroyer."`   | Short one-liner                    |
| `difficulty`| `Beginner`                   | Beginner / Intermediate / Advanced |
| `type`      | `Mutant`                     | Beast / Alien / Mutant / Robot     |
| `game`      | `- Tokyo`                    | Which game the monster is from     |
| `style`     | `- Smasher`                  | One or more from the styles list   |
| `stats`     | see example above            | aggression / resilience / energy / speed, 1-5 |

### Suggested Markdown body sections

- `## Overview` — who the monster is and what makes them unique
- `## Playstyle` — how they win, based on their evolutions
- `## Evolution Cards` — one `###` section per card with blockquote for official text
- `## Tips & Strategies` — with `### Early Game` and `### Late Game`
- `## Best Evolution Combos` — key card combinations
- `## Card Priorities` — numbered list of recommended base market cards
- `## Common Mistakes` — bullet list of things to avoid

## Contributing

This guide is open source and community driven. Contributions are very welcome.

If you want to help out:

- **Found an error?** Open an issue or submit a pull request on [GitHub](https://github.com/kingofguide/tokyo)
- **Want to add a monster?** Copy an existing `.md` file, follow the frontmatter format above, write the guide, and submit a pull request
- **Want to improve an existing guide?** Edit the relevant `.md` file and submit a pull request

### We especially need help with

- Guides for monsters not yet covered
- Evolution card text verification against physical cards
- Playstyle and strategy input from experienced players
- Spotting errors or outdated information

All contributors will be credited. If you're not familiar with GitHub but have knowledge to share, feel free to open an issue and share your notes there instead.
